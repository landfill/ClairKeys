## Why this exists

`Security Audit` (`npm audit --audit-level high`, `.github/workflows/test.yml:173` and
`.github/workflows/pr-checks.yml:213`) is a required status check, so a red gate blocks the merge
button for every PR. It went red on `main` at documentation-only commit `f39fbb6` while `Run Tests`,
`Lint`, `E2E Tests`, `Build application`, and `Test before deploy` all passed.

No dependency changed between the green `1e3d515` (2026-07-22) and `f39fbb6`. `npm audit` grades
against the advisory database at run time, so newly published advisories flipped a tree that had not
moved. `docs/recovery/reviews/PR-25.md` had already recorded next-auth/postcss/uuid sitting at
moderate after the `sharp` pin; those entries escalated rather than appearing from nowhere.

This is the second occurrence of the same failure mode as PR #25 and it will recur. It is a property
of the gate, not of any one dependency.

## The three problems and why each got a different answer

| Package | Severity at gate | Answer | Reasoning |

| `next-auth` 4.24.14 | critical ×3 — GHSA-xmf8-cvqr-rfgj (uncaught exception on malformed Bearer headers), GHSA-7rqj-j65f-68wh (email normalizer validates before Unicode normalization, homoglyph `@` bypass), GHSA-x445-f3h2-j279 (OAuth state/nonce/PKCE cookies not bound to the issuing provider) | upgrade to 4.24.15 | Patch already inside the declared `^4.24.11` range; no range change needed |

The top level already resolved 8.5.x via `@tailwindcss/postcss`, so the pin converges the tree instead of forking it. The project has no direct postcss dependency, matching PR #25's `sharp` precedent |
| `next` 15.5.20 | high ×8 — SSRF in Server Actions (GHSA-89xv-2m56-2m9x) and in rewrites (GHSA-p9j2-gv94-2wf4), cache confusion (GHSA-68g3-v927-f742, GHSA-4633-3j49-mh5q), DoS via Server Actions (GHSA-m99w-x7hq-7vfj) and Image Optimization SVGs (GHSA-q8wf-6r8g-63ch), unbounded Edge Server Action payload (GHSA-4c39-4ccg-62r3), unauthenticated Server Function endpoint disclosure (GHSA-955p-x3mx-jcvp) | bump exact pin to 15.5.21 | npm reported the fix as "outside the stated dependency range" only because `package.json` pins `next` exactly. `fixAvailable.isSemVerMajor` is `false` — this is a patch bump within 15.5.x |

Note the ordering effect: `next` only surfaced its own eight high advisories after the postcss
override removed the inherited postcss finding. The first audit run attributed a single high entry to
`next` via postcss and masked the rest.

### Deliberately not changed

`uuid@8.3.2` nested under `next-auth` (GHSA-w5hq-g745-h8pq, missing buffer bounds check in v3/v5/v6)
is **moderate**, so it never reached the `--audit-level high` gate. An `overrides` pin to `>=11.1.1`
would have forced next-auth from uuid v8's default export onto v11's named exports to fix something
the gate does not test. The `next-auth` upgrade cleared the entry anyway; the final tree reports zero
vulnerabilities at every severity.
