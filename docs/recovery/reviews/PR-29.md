## Why this exists

Every commit on `main` ended red because four jobs had never once succeeded. The repository has **no
secrets configured at all** ‚Äî `gh secret list` and
`gh api repos/landfill/ClairKeys/environments/production/secrets` both return empty:

| Job | Failure |
|---|---|
| `Deploy to production` | `amondnet/vercel-action@v25` given an empty `secrets.VERCEL_TOKEN` |
| `Run database migrations` | `prisma migrate deploy` given an empty `DATABASE_URL` ‚Üí Prisma `P1012` |
| `Notify deployment status` | its failure branch `exit 1`s when either of the above fails |
| `Post-deploy health check` | permanently `skipped` behind `needs: [deploy, database-migrate]` |

The permanent red is the point, not the cosmetics. When every commit fails identically, a real
failure carries no signal ‚Äî which is why production served a pre-P0 bundle for days, with issue #18's
10-second audio cutoff reproducing live, while CI looked exactly as red as usual. That is recorded in
`docs/recovery/validation/2026-07-24-production-serves-pre-p0-bundle.md`.

## The rename, and why it is in scope

A workflow named `Deploy` that only runs tests is worse than no workflow: in the checks list, and in
the next session's reading of it, it presents as evidence that a merge shipped. That is precisely the
misreading this investigation existed to correct, so leaving the name would rebuild the trap.

New job names deliberately avoid `Run Tests` and `Build Check`, which are check names already owned by
`test.yml` and `pr-checks.yml`. `Run Tests` is one of the four required status checks, so a duplicate
name there would be actively dangerous.

## Left alone deliberately

The surviving `build` job still passes `secrets.DATABASE_URL`, `NEXTAUTH_SECRET`, `SUPABASE_URL`, and
`SUPABASE_ANON_KEY` to `npm run build`. All resolve to empty strings today and the job passes anyway,
so the build does not need them. That job was never failing and sits outside issue #28's scope. An
early draft of the test asserted their absence and failed ‚Äî correctly, since the assertion was untrue.
The test file now records the reasoning inline instead.

## CI and review status

### The checks did not run at first ‚Äî and did not fail either

After the branch was pushed, `gh run list --branch codex/chore-deploy-workflow` returned **zero
runs** and `gh pr checks 29` listed only the two Vercel entries. No required check was red; none
existed at all.

Cause: `test.yml` and `pr-checks.yml` trigger on `pull_request`, and `test.yml`'s `push` trigger is
limited to `[main, develop]`, so a work-branch push alone never starts them. The PR's opening event
appears to have been dropped ‚Äî the first `git push` of this branch was rejected with a GitHub
`500 Internal Server Error` and only succeeded on retry, and an earlier `git push --delete` in the
same session hit a `500` as well. GitHub Actions was healthy throughout: pushes to `main` kept
producing runs normally.

Remedy: `gh pr close 29 && gh pr reopen 29`. `pr-checks.yml` lists `reopened` in its `types`, and
`test.yml` has no `types` filter (so it defaults to include `reopened`). Both workflows queued
immediately.

**Worth carrying forward:** a missing required check is not visually distinct from a check that has
not started yet, and `gh pr checks` reporting "all pass" can mean "the only checks present are
Vercel's". An agent waiting on CI must wait for the four required checks to *appear*, not merely for
nothing to be pending ‚Äî a first attempt here used the weaker condition and reported success off two
Vercel entries.

### CodeRabbit was rate limited first

The initial review never started: the check read `pass` with the status text **"Review rate limited"**,
and the comment body said `@landfill` had reached the per-developer PR review limit with the next
review available in 15 minutes. This is an adaptive rolling-window limit on the Pro Plus plan, not a
free-tier quota; opening PR #27 and #29 in quick succession tripped it.

A `pass` check that means "the review never ran" is indistinguishable at a glance from a `pass` that
means "reviewed, nothing found" ‚Äî the same shape of misread as the green `Vercel` check that only ever
meant a Preview build. The remedy was to wait out the window and post `@coderabbitai review`, after
which the existing comment was rewritten in place with the real result.
